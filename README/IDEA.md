# Mobile App Architecture Documentation (Auth & Video Player)

## 1. Authentication

### 1.1. Overview

Authentication is handled using Supabase Auth, supporting Google Sign-In (Android/iOS) and Apple Sign-In (iOS). The user's session is persisted across app restarts using `AsyncStorage`. Global authentication state (session, user, loading status) is managed using Jotai.

### 1.2. Core Components & Files

- **Supabase Client (`mobile/utils/supabase.ts`):** Initializes and exports the Supabase client instance configured with environment variables (`EXPO_PUBLIC_SUPABASE_URL`, `EXPO_PUBLIC_SUPABASE_ANON_KEY`).
- **Auth UI Component (`mobile/components/Auth.native.tsx`):** Provides platform-specific sign-in buttons (Google, Apple) and handles the respective sign-in flows using `expo-apple-authentication` and `@react-native-google-signin/google-signin`. It interacts with `supabase.auth.signInWithIdToken` upon successful retrieval of ID tokens. Includes an `onAuthSuccess` callback.
- **Auth Utilities (`mobile/utils/auth.ts`):**
  - `createAuthenticatedUser`: (Currently unused in this flow, potentially for creating profile records after signup).
  - `signOut`: Signs the user out of Supabase, clears the Jotai user state (via `clearUserData` placeholder), removes relevant `AsyncStorage` items (including session and onboarding status), and redirects to the `/onboarding` screen.
- **Jotai Auth State (`mobile/store/authAtom.ts`):** Defines the `authAtom`, a Jotai atom holding the `AuthState` interface (`session`, `user`, `loading`). Initial state has `loading: true`.
- **Jotai Store (`mobile/store/index.ts`):** Creates and exports the Jotai store instance.
- **User Slice Placeholder (`mobile/store/userSlice.ts`):** Contains a placeholder `clearUserData` function, used by `signOut`. In a full Jotai implementation, this might be replaced by directly resetting atoms within `signOut` or via a dedicated reset utility.
- **Auth Hook (`mobile/hooks/useAuth.ts`):** Provides a convenient `useAuth` hook to access the current auth state (`session`, `user`, `userId`, `isLoading`, `isAuthenticated`) from the `authAtom`.
- **Root Layout (`mobile/app/_layout.tsx`):**
  - Initializes Supabase auth listener (`onAuthStateChange`).
  - On app start, attempts to load the session from `AsyncStorage` (`SUPABASE_SESSION_KEY`).
  - Validates the loaded session's expiration.
  - Sets the initial state of `authAtom`.
  - The `onAuthStateChange` listener updates `authAtom` and persists the session to `AsyncStorage` on `SIGNED_IN`/`TOKEN_REFRESHED` events, clearing it on `SIGNED_OUT`.
  - Manages the splash screen visibility based on app readiness (`appReady` state).
- **Onboarding Screen (`mobile/app/onboarding.tsx`):**
  - Uses the `useAuth` hook.
  - Displays loading text while `isLoading` is true.
  - If the user is not authenticated (`!isAuthenticated`), it renders the `Auth.native.tsx` component.
  - Uses `useEffect` to automatically redirect (`router.replace`) to the main app (e.g., `/(tabs)/search`) once `isAuthenticated` becomes true.

### 1.3. Authentication Flow

1.  App starts, `_layout.tsx` initializes.
2.  `initializeAuth` runs: tries to load session from `AsyncStorage`.
3.  Sets initial `authAtom` state (user might be null or loaded from storage). `loading` becomes `false`.
4.  Supabase `onAuthStateChange` listener is attached.
5.  App UI renders. If `isAuthenticated` is false (and `isLoading` is false), the user might be directed to `/onboarding`.
6.  On the `/onboarding` screen, the `Auth` component is displayed.
7.  User clicks Google/Apple Sign-In.
8.  `Auth.native.tsx` handles the native sign-in flow.
9.  On success, obtains an ID token.
10. Calls `supabase.auth.signInWithIdToken` with the provider and token.
11. Supabase verifies the token and establishes a session.
12. The `onAuthStateChange` listener in `_layout.tsx` fires with the `SIGNED_IN` event and the new session.
13. The listener updates the `authAtom` with the session/user.
14. The listener saves the session JSON to `AsyncStorage`.
15. The `useEffect` hook in `/onboarding` detects `isAuthenticated` is now true and navigates the user away.
16. On subsequent app opens, the session is loaded directly from `AsyncStorage`, skipping the login step.
17. Sign-out clears the session, `AsyncStorage`, Jotai state, and redirects to `/onboarding`.

## 2. Video Player & On-the-Fly Audio Generation

### 2.1. Overview

The video player screen (`mobile/app/video/[id].tsx`) plays a YouTube video using `react-native-webview`. It implements on-the-fly audio generation and playback for dubbed audio based on transcription segments fetched from the server. It interacts with server actions via API helper functions.

### 2.2. Core Components & Files

- **Video Player Screen (`mobile/app/video/[id].tsx`):**
  - Gets `youtubeVideoId`, `language`, `voice`, `position` from route params (`useLocalSearchParams`).
  - Uses `react-native-webview` to embed the YouTube IFrame Player.
  - Uses `expo-av` (`Audio.Sound`) for playing generated audio chunks (`soundRef`).
  - Manages various states: `loadingState`, `errorMessage`, `playerState`, `currentTime`, `videoDuration`, `isFavorite`, `videoTitle`, `dbVideoId`, `downloadJobId`, `transcription`, `language`, `voice`, `generatedChunks`, `audioStatusText`, `isAudioGenerating`, `currentlyPlayingChunkKey`.
  - Uses the `useAuth` hook to get `userId` and `isAuthenticated` status.
  - **Workflow Functions:**
    - `initiateVideoProcessing`: Called on mount. Sends video URL and `userId` to `startVideoProcessingApi`. Sets `dbVideoId`, `downloadJobId`, triggers `checkFavoriteStatus`, and sets `loadingState` to 'downloading'.
    - `startJobPolling`: Initiates polling for 'download' or 'transcription' status using `getJobStatusApi` or `getTranscriptionApi` (currently mocks).
    - `handleDownloadComplete`: Called when download polling succeeds. Calls `startTranscriptionApi`. Sets `loadingState` to 'transcribing'.
    - `handleTranscriptionComplete`: Called when transcription polling succeeds. Stores transcription segments, sets `loadingState` to 'ready', seeks player if needed, triggers initial audio generation.
    - `manageAudioGeneration`: Determines the next needed audio segment based on `currentTime` + `LOOKAHEAD_SECONDS`, checks `generatedChunks`, calls `generateAudioChunkApi` if needed.
    - `manageAudioPlayback`: Finds the current transcription segment based on `currentTime`, checks `generatedChunks` for the corresponding audio, calls `loadAndPlayAudio` to load/switch the `Audio.Sound` instance.
    - `loadAndPlayAudio`: Unloads previous sound, creates a new `Audio.Sound` instance with the chunk URL, seeks to the correct position within the chunk based on `currentTime` and segment start time.
    - `checkFavoriteStatus`: Calls `getFavoriteStatusApi` with video/user details to update `isFavorite` state.
    - `handleToggleFavorite`: Optimistically updates `isFavorite` state, calls `toggleFavoriteApi`. Reverts state on API failure.
    - `updateWatchHistory`: Calls `updateHistoryApi` with video ID, position, and `userId` when the player is paused/ended or the app goes to the background.
  - **UI:** Renders loading indicators, error messages, the `WebView`, video title, audio status, and header controls (back, title, favorite toggle).
- **API Library (`mobile/lib/api.ts`):**
  - Contains functions (`startVideoProcessingApi`, `startTranscriptionApi`, `getJobStatusApi`, `getTranscriptionApi`, `generateAudioChunkApi`, `updateHistoryApi`, `toggleFavoriteApi`, `getFavoriteStatusApi`) that act as wrappers for calling server actions.
  - Includes a `callServerAction` helper function using `fetch` to POST JSON data to assumed server endpoints (`/api/actions/[actionName]`). Handles basic success/error checking and maps results to `ActionResponse`.
  - Defines TypeScript interfaces for the Input and Output of each action.
  - **Currently:** Most functions use `callServerAction`. `getJobStatusApi` and `getTranscriptionApi` still use mock implementations pending decisions on their server-side implementation (POST action vs. dedicated GET endpoint). `getFavoriteStatusApi` also uses a mock.
- **Shared Types (`mobile/types/`):**
  - `actions.ts`: Copied from the server. Defines `ActionResponse`, `AppError`, `AppErrorCode`, `appErrors`.
  - `serverActions.ts`: Defines input/output interfaces specific to the server actions (e.g., `StartVideoProcessingInput`, `GenerateAudioChunkOutput`).
  - `errors.ts`: (Not currently used, errors are defined in `actions.ts`).

### 2.3. Video/Audio Workflow

1.  Screen mounts, `initiateVideoProcessing` sends videoId, URL, userId to the server. Server starts download job, returns `jobId` and internal `videoId`.
2.  Client polls `getJobStatusApi` using `jobId`.
3.  Once download completes, client calls `startTranscriptionApi` (server starts transcription job).
4.  Client polls `getTranscriptionApi` using `videoId`.
5.  Once transcription completes, client receives transcription segments. `loadingState` becomes 'ready'.
6.  `manageAudioGeneration` is called. It looks ahead (`LOOKAHEAD_SECONDS`) from `currentTime` (initially 0 or `initialPosition`), finds the first needed segment, and calls `generateAudioChunkApi` with videoId, language, voice, and segment times.
7.  Server generates the audio chunk, saves it, returns a public URL (`publicUrl`) and `storagePath`.
8.  Client stores the chunk info (`GenerateAudioChunkOutput`) in the `generatedChunks` map, keyed by a unique identifier derived from videoId, lang, voice, and timestamps.
9.  As `currentTime` updates (via interval while playing):
    - `manageAudioPlayback` identifies the `currentSegment`.
    - If the `chunkKey` for `currentSegment` is different from `currentlyPlayingChunkKey`:
      - It looks up the `chunkKey` in `generatedChunks`.
      - If found, calls `loadAndPlayAudio` with the `publicUrl`. The sound is loaded, potentially seeked, and played (if the video player is playing). `currentlyPlayingChunkKey` is updated.
      - If not found, triggers `manageAudioGeneration` to request the chunk.
    - `manageAudioGeneration` identifies the _next_ needed chunk based on `currentTime + LOOKAHEAD_SECONDS` and requests it via `generateAudioChunkApi` if not already present in `generatedChunks`.
10. Seeking (`handleSeek`) resets `generatedChunks`, unloads the current sound, and immediately triggers `manageAudioGeneration(true)` and `manageAudioPlayback` to fetch/play audio for the new position.
